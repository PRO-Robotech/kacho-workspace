# Kachō Architecture Vault — ребилд как живая 5-уровневая модель

**Дата:** 2026-05-22
**Статус:** design — ожидает ревью
**Тип:** cross-repo feature (KAC-эпик)
**Затронутые репо:** `kacho-corelib` (инструмент), все `kacho-<svc>`, `kacho-workspace` (агрегация)

## 1. Проблема

Obsidian-vault `kacho-workspace/obsidian/kacho/` (259 файлов) деградировал и
не отражает реальность кода. Диагностика (Explore, 2026-05-22):

- **Покрытие ~42%** — домен `compute` (20 сервисов) и `loadbalancer` не
  документированы вовсе; в `packages/iam-domain.md` нет сущностей Phase 6–8.
- **Мёртвые ссылки** — каталог `kacho-resource-manager/` живёт со `status: stable`,
  хотя репо упразднён в KAC-124.
- **Сломанная семантика** — все 42 `rpc/*.md` стоят `status: planned`, хотя
  бэкенды реализованы; `methods_count` захардкожен во frontmatter и врёт при
  изменении proto.
- **Нет привязки к коду** — frontmatter ссылается на `proto_file`, но без
  commit/SHA → невозможно сказать, на какой версии кода заметка ещё верна.

Корень: vault написан вручную и дрейфует. Поскольку код пишет ИИ, **нет гарантии,
что в кодовую базу не попал мёртвый код или функциональность, которая никогда не
будет использована** — а ручной vault это не ловит и сам отстаёт на 1–3 месяца.

## 2. Цель

Пересобрать vault как **живую модель того, как работает продукт** — с чётким
понятием «как устроено» на каждом уровне детализации, и с **CI-гарантией**, что
модель не расходится с кодом и что код не содержит недокументированной или
мёртвой функциональности.

## 3. Модель — вертикаль из 5 уровней

C4-аналогичная вертикальная декомпозиция. Никаких отдельных категорий
(`edges/` / `resources/` / `rpc/`) — то, что было «фасетами», растворено в
**секциях внутри уровней**.

| Ур. | Имя | C4-аналог | Содержимое | Режим |
|---|---|---|---|---|
| L0 | Проект | System Context | Платформа Kachō: домены, 9 репо, cross-service рантайм-граф (секция «Связи») | курируется |
| L1 | Приклад | Container | Один сервис: зона ответственности; секция «Контракт» (полный RPC-список); секция «Связи» (кого зовёт / кто зовёт его) | курируется |
| L2 | Функциональность | Component | Фича-кластер / use-case-группа («Network lifecycle», «IPAM allocation») + якорные entry-points; курируемое «зачем / lifecycle / gotchas» ресурсов, которыми владеет фича | курируется |
| L3 | Функции | Code | Call-граф от якорей, exported-сигнатуры, reachability | **генерится** |
| L4 | Переменные | (sub-Code) | Domain-типы, поля, DB-колонки, config-ключи, константы | **генерится** |

**Граница режимов:** L0–L2 курируются (это «зачем» — пишет человек/Claude),
L3–L4 генерятся из кода (это «как» — по построению не врут и не содержат
мёртвого кода).

**Растворение бывших фасетов:**
- *Контракт* (proto/RPC) → секция L1 (полный RPC-список сервиса) + якоря в L2.
- *Связи* (cross-service рёбра) → секция L0 (глобальный граф) + секция L1 (рёбра
  этого приклада); владелец ребра = вызывающий приклад.
- *Данные* → курируемое «зачем/lifecycle/gotchas» ресурса → L2; генерённые
  таблицы полей/колонок/типов → L4.

**Гранулярность L2:** одна заметка = связный фича-кластер (~30–50 заметок на
проект), якорь — набор RPC/entry-points. Не «один RPC = одна заметка»
(мелко, рвёт narrative) и не «один ресурс = одна заметка» (кросс-ресурсные
фичи как reconciler/outbox/IPAM остались бы бездомными).

## 4. Traceability-хребет и анти-dead-code механизм

### 4.1 Frontmatter L2-заметки

```yaml
---
level: functionality
repo: kacho-vpc
anchors:                              # entry-points, которыми владеет фича
  - rpc: vpc.v1.NetworkService/Create
  - rpc: vpc.v1.NetworkService/Delete
  - worker: NetworkOutboxWorker
status: implemented                   # ВЫЧИСЛЯЕТСЯ инструментом, не печатается руками
source_sha: <sha последнего ревью кода под этой заметкой>
---
```

### 4.2 Инструмент `archgraph`

Маленькая Go-программа, живёт в `kacho-corelib` (переиспользуется всеми
сервисами), запускается в каждом репо. Использует `golang.org/x/tools/go/packages`
+ `golang.org/x/tools/go/callgraph` (либо обёртку над `golang.org/x/tools/cmd/deadcode`).

Алгоритм:

1. Парсит `main`-пакеты репо → перечисляет **entry-points**: зарегистрированные
   gRPC-методы (распознаётся вызов `RegisterXxxServer`), фоновые воркеры,
   реконсилеры, cron-задачи.
2. Считает reachability (call-граф) от каждого entry-point.
3. Прогоняет три детерминированные проверки (см. 4.3).
4. Генерит L3/L4-артефакты в `docs/arch/generated/`: на каждую функциональность —
   call-дерево + сигнатуры; на приклад — таблицы domain-типов / полей / DB-колонок
   / config-ключей.
5. Вычисляет и пишет обратно `status` в L2-заметки.

### 4.3 Три CI-блокирующие проверки

- **C1 — Полнота.** Каждый entry-point, достижимый из `main`, обязан быть заявлен
  ровно в одном L2-`anchors`. Незаявленный entry-point → **FAIL** «недокументированная
  функциональность — опиши её в L2-заметке или удали». Якорь, указывающий на
  несуществующий entry-point → **FAIL** «протухшая заметка».
- **C2 — Мёртвый код.** Любой exported-символ, не достижимый ни от одного
  entry-point → **FAIL**. Жёстко. Единственное исключение — явная аннотация в коде
  `// archgraph:keep <причина>` (намеренное публичное API, тест-хелперы).
- **C3 — Свежесть.** Для каждой курируемой заметки хешируется reachable-set
  файлов её якорей; хеш ≠ `source_sha` → **FAIL** «код под заметкой изменился —
  перечитай заметку и обнови `source_sha`». Проверка детерминированная: не
  проверяет правильность прозы, а форсит факт перечитки после изменения кода.

Все три блокируют CI. Итог: ИИ физически не смёржит (a) мёртвый код, (b) новый
RPC без объявления его намеренной функциональностью, (c) изменение кода под
заметкой без перечитки этой заметки.

### 4.4 `status` — вычисляемый, не печатаемый

Фикс найденного бага «`status: planned` врёт у всех 42 заметок». `status`
вычисляет `archgraph` и пишет обратно:
- `implemented` — все якоря существуют и достижимы;
- `partial` — часть якорей отсутствует в коде;
- `planned` — заметка есть, якорей в коде ноль.

Руками не печатается никогда; `git diff --exit-code` ловит расхождение.

## 5. Топология и хранение

- Сгенерённый низ (L3/L4) **и** курируемый верх (L1/L2) живут в `docs/arch/`
  каждого сервисного репо рядом с кодом.
- `kacho-workspace` хранит нативно: L0 Проект-заметку и `KAC/`-трейл (это не
  код-производное).
- Agregация: `kacho-workspace` получает `make vault-sync` — для каждого репо в
  `project/` копирует `<repo>/docs/arch/**` → `obsidian/kacho/<repo>/`.
  Obsidian-vault становится **read-only агрегатом**.

## 6. CI

### 6.1 Сервисное репо — job `arch-vault` (всё блокирующее)

```
make arch-gen          # регенерит L3/L4 → docs/arch/generated/
git diff --exit-code   # дрейф: генерённый слой протух относительно кода
make arch-audit        # C1 полнота + C2 мёртвый код + C3 свежесть
```

### 6.2 Workspace — cross-repo целостность связей

Ребро `vpc→compute` нельзя проверить внутри одного репо. Workspace-CI после
`vault-sync` валидирует `target:` каждого ребра против сгенерённого инвентаря
entry-points целевого репо. Advisory на PR сервиса, блокирует на workspace-sync.

### 6.3 Локальный хук

Существующий `vault-stop-check.sh` (Stop-хук) расширяется вызовом
`make arch-audit` — Claude видит фейлы до пуша.

## 7. Стратегия ребилда

Полный структурный ребилд, «снизу вверх», staged:

1. **Регенерация L3/L4** во всех репо — детерминированно, мгновенно, точно.
   Это новый пол.
2. **Авто-сид скелетов L1/L2** — `archgraph` на каждый кластер entry-points
   (сервис + ресурс) эмитит заготовку L2-заметки: предзаполненные `anchors`,
   вычисленный `status`, тело `TODO`.
3. **Claude заполняет narrative** по скелетам — единственный дорогой шаг,
   ограничен (~30–50 заметок/проект, скоупится по репо, параллелится через
   dispatching-parallel-agents).
4. **Миграция знания** из старого vault — настоящие gotchas / history / KAC-trail
   Claude переносит в новые L1/L2, **не выбрасывает**.
5. **Удаление** старого протухшего дерева (`kacho-resource-manager/`, 259 ручных
   файлов) — после замещения.

**Выбранный подход:** генерим скелеты → Claude дописывает. Отвергнутая
альтернатива: Claude читает код и пишет всё с нуля (медленно, галлюцинации, нет
детерминированного пола).

## 8. Раскатка (KAC-эпик)

Топологический порядок (cross-репо граф зависимостей):

1. `kacho-corelib` — инструмент `archgraph` + общие Makefile-цели
   (`arch-gen` / `arch-audit`).
2. По одному сервису за раз: `kacho-vpc` → `kacho-iam` → `kacho-compute` → …
   Для каждого: регенерация L3/L4, авто-сид + заполнение L1/L2, новый CI-job,
   удаление старых vault-заметок этого домена.
3. `kacho-workspace` — `make vault-sync`, L0 Проект-заметка, workspace-CI
   cross-repo проверки, удаление остатков старого дерева.

Эпик заводится в YouTrack `KAC` с Subtask-декомпозицией по этому порядку;
ветки `KAC-<N>` в каждом затронутом репо.

## 9. Definition of Done

- [ ] `archgraph` в `kacho-corelib` — реализован, с integration-тестами
      (testcontainers не нужен; тесты на синтетических Go-пакетах).
- [ ] Каждый сервисный репо: job `arch-vault` зелёный, C1/C2/C3 активны и
      блокируют.
- [ ] L3/L4 сгенерированы и закоммичены для всех сервисов; `git diff` чист.
- [ ] L1/L2 narrative заполнен для всех ~30–50 функциональностей на сервис;
      `status` вычислен, не захардкожен.
- [ ] Знание из старого vault (gotchas/history/KAC-trail) мигрировано.
- [ ] Старое дерево (`kacho-resource-manager/` и протухшие ручные файлы)
      удалено.
- [ ] `make vault-sync` в `kacho-workspace`, Obsidian-vault — read-only агрегат.
- [ ] Workspace-CI cross-repo проверка связей зелёная.
- [ ] `vault-stop-check.sh` расширен `make arch-audit`.

## 10. Открытые вопросы / риски

- **Точность call-графа.** CHA/RTA-анализ может пере- или недооценить
  достижимость при тяжёлом использовании интерфейсов / reflection. `archgraph`
  должен выбрать алгоритм (RTA как стартовая точка) и иметь `archgraph:keep` как
  предохранитель для ложных C2-срабатываний. Калибруется на первом сервисе
  (`kacho-vpc`).
- **Распознавание entry-points.** gRPC-методы детектятся по `RegisterXxxServer`;
  воркеры/реконсилеры — по конвенции (конструкторы в `cmd/<svc>/main.go`). Если
  конвенция в репо не выдержана — C1 даст ложные срабатывания; первый сервис
  калибрует список паттернов.
- **Боль параллельных агентов.** Жёсткий блокирующий гейт при параллельной
  работе нескольких агентов над одним репо может ронять CI часто. Принято
  осознанно (запрошен максимум гарантий); смягчается тем, что `arch-audit`
  доступен локально через Stop-хук.
