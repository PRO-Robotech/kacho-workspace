# WS-2.3 — AuthZ Decision-Cache Invalidation on Grant/Revoke — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make `AccessBinding.Create`/`Delete` invalidate the api-gateway authz decision-cache so a revoked grant stops authorising within ≤1 request instead of lingering up to the 5 s cache TTL — unblocking the `AUTHZ-REVOKE-ENFORCED-A-NOB` newman e2e (currently RED, blocks `kacho-iam` PR #17).

**Architecture:** Two-channel invalidation. (1) **Synchronous self-flush** — the gateway proxies every `POST/DELETE /iam/v1/accessBindings*` itself; on a 2xx response it flushes its own `decisionCache` immediately (deterministic for the replica in the request path; this is what makes the e2e pass). (2) **Cross-replica outbox drain** — `kacho-iam` writes a `subject_change_outbox` row inside the same DB transaction as the binding mutation; a new internal RPC `InternalIAMService.PollSubjectChanges` exposes that outbox by cursor; every gateway replica runs a background poll-loop that drains it and flushes its cache, so sibling replicas (HPA scales gateway to 10) converge within one poll interval. No new DB credentials on the gateway — the poll reuses the existing `iamInternal` gRPC connection.

**Tech Stack:** Go 1.22, gRPC + grpc-gateway, pgx/v5, sqlc-free handwritten SQL, OpenFGA, buf/protoc, testcontainers-go (Postgres 16), Newman (Postman) e2e, goose migrations.

**Why not pure LISTEN/NOTIFY:** the `subject_change_outbox` NOTIFY trigger (`kacho_iam_subject_outbox_added`) already exists, but the gateway has no Postgres connectivity (`pgx` is only an indirect dep) and is the most exposed component — handing it `kacho_iam` DB credentials widens blast radius. The RPC-poll reuses an existing trusted gRPC channel. Determinism for the e2e (which revokes then re-reads 0.37 s later, faster than any sane poll interval) is provided by the synchronous self-flush, not the poll.

## Background — confirmed root cause

`AUTHZ-REVOKE-ENFORCED-A-NOB` (in `kacho-iam/tests/newman/cases/authz-deny.py`) fails with:

```
[REVOKE] #8: after revoke NOB is DENIED project-A1 (403)
revoke did NOT remove the FGA tuple — access still live:
{"code":5,"message":"Project prj... not found"}: expected 404 to equal 403
```

Sequence (steps 0.37 s apart): step 3 `nob-get-project-A1` → gateway runs an FGA Check → caches `ALLOW` in `decisionCache` (LRU 10k, TTL 5 s, `kacho-api-gateway/internal/middleware/authz.go`). Step 4-5 revoke → FGA tuple **is** deleted (service log: `openfga revoke tuples deleted count 2`). Step 6 `nob-get-project-A1` (+0.37 s) → **cache HIT** on stale `ALLOW` → request forwarded to `ProjectService.Get` → that service's owner-only check returns `404` (NOB is not the project owner) → test expected `403`.

Confirmed dead scaffolding:
- `kacho-iam` migration `0002_fga_outbox.sql` created `subject_change_outbox` (cols `id, subject_id, op, created_at, notified_at`; `op ∈ {binding_upsert, binding_delete, group_member_change}`) + trigger `subject_change_outbox_notify` → `pg_notify('kacho_iam_subject_outbox_added', id)`. **No code writes to it; no code drains it.**
- `kacho-api-gateway/internal/middleware/authz.go` has `decisionCache.Invalidate()` (full flush) and `InvalidateSubject(prefix)` — **zero callers**.

This plan wires both dead ends together.

```go
// subject_change_service.go — WS-2.3 read-side of subject_change_outbox.
// Exposes the outbox by ascending-id cursor for api-gateway authz-cache
// invalidation. Read-only; no mutation.
package service

import (
	"context"
	"fmt"

	iamv1 "github.com/PRO-Robotech/kacho-proto/gen/go/kacho/cloud/iam/v1"
	"github.com/jackc/pgx/v5/pgxpool"
)

type SubjectChangeService struct{ pool *pgxpool.Pool }

func NewSubjectChangeService(pool *pgxpool.Pool) *SubjectChangeService {
	return &SubjectChangeService{pool: pool}
}

func (s *SubjectChangeService) PollSubjectChanges(ctx context.Context, sinceID int64, limit int32) (*iamv1.PollSubjectChangesResponse, error) {
	if limit <= 0 {
		limit = 256
	}
	if limit > 1000 {
		limit = 1000
	}
	rows, err := s.pool.Query(ctx,
		`SELECT id, subject_id, op FROM kacho_iam.subject_change_outbox
		  WHERE id > $1 ORDER BY id ASC LIMIT $2`, sinceID, limit)
	if err != nil {
		return nil, fmt.Errorf("poll subject_change_outbox: %w", err)
	}
	defer rows.Close()

	out := &iamv1.PollSubjectChangesResponse{}
	for rows.Next() {
		var c iamv1.SubjectChange
		if err := rows.Scan(&c.Id, &c.SubjectId, &c.Op); err != nil {
			return nil, fmt.Errorf("scan subject_change: %w", err)
		}
		out.Changes = append(out.Changes, &c)
	}
	if err := rows.Err(); err != nil {
		return nil, fmt.Errorf("iterate subject_change: %w", err)
	}
	if err := s.pool.QueryRow(ctx,
		`SELECT COALESCE(MAX(id), 0) FROM kacho_iam.subject_change_outbox`).
		Scan(&out.HeadId); err != nil {
		return nil, fmt.Errorf("head_id subject_change_outbox: %w", err)
	}
	return out, nil
}
```

```go
// subjectChangingFQNs — gRPC FQNs whose success changes a subject's grants.
// On a 2xx response the gateway flushes its decision cache so the new grant
// state takes effect immediately for this replica (WS-2.3 self-flush). Sibling
// replicas converge via the subject-change poll-loop.
var subjectChangingFQNs = map[string]struct{}{
	"kacho.cloud.iam.v1.AccessBindingService/Create": {},
	"kacho.cloud.iam.v1.AccessBindingService/Delete": {},
}

// MaybeFlushOnMutation flushes the decision cache when fqn is a grant-changing
// RPC and the proxied response was successful (2xx / gRPC OK==200 here, since
// the HTTP layer reports HTTP status). Safe to call on every request.
func (m *AuthzMiddleware) MaybeFlushOnMutation(fqn string, httpStatus int) {
	if m.cache == nil || httpStatus < 200 || httpStatus >= 300 {
		return
	}
	if _, ok := subjectChangingFQNs[normalizeFQN(fqn)]; !ok {
		return
	}
	m.cache.Invalidate()
	m.cfg.Logger.Info("authz decision-cache flushed on grant mutation", "fqn", fqn)
}

// Cache exposes the decision cache so the subject-change watcher can flush it.
func (m *AuthzMiddleware) Cache() *decisionCache { return m.cache }
```

```go
// subject_change_watcher.go — WS-2.3 cross-replica authz-cache invalidation.
// Polls kacho-iam InternalIAMService.PollSubjectChanges by ascending-id cursor;
// on any non-empty batch, flushes this replica's authz decision cache. The
// request-path replica already self-flushed (see authz.MaybeFlushOnMutation);
// this loop converges sibling replicas within one interval.
package watcher

import (
	"context"
	"log/slog"
	"time"
)

// Poller — narrow port over the PollSubjectChanges RPC.
type Poller interface {
	PollSubjectChanges(ctx context.Context, since int64) (ids []int64, headID int64, err error)
}

type SubjectChangeWatcher struct {
	poller   Poller
	flush    func()
	interval time.Duration
	logger   *slog.Logger
	cursor   int64
	primed   bool
}

func New(p Poller, flush func(), interval time.Duration, logger *slog.Logger) *SubjectChangeWatcher {
	if interval <= 0 {
		interval = 2 * time.Second
	}
	return &SubjectChangeWatcher{poller: p, flush: flush, interval: interval, logger: logger}
}

// Run blocks until ctx is cancelled.
func (w *SubjectChangeWatcher) Run(ctx context.Context) {
	t := time.NewTicker(w.interval)
	defer t.Stop()
	for {
		select {
		case <-ctx.Done():
			return
		case <-t.C:
			w.tick(ctx)
		}
	}
}

func (w *SubjectChangeWatcher) tick(ctx context.Context) {
	ids, headID, err := w.poller.PollSubjectChanges(ctx, w.cursor)
	if err != nil {
		w.logger.Warn("subject-change poll failed", "err", err)
		return
	}
	// First successful poll on a fresh gateway: adopt head as cursor without
	// flushing — the cache is empty/cold at startup anyway.
	if !w.primed {
		w.primed = true
		w.cursor = headID
		if len(ids) == 0 {
			return
		}
	}
	if len(ids) == 0 {
		return
	}
	for _, id := range ids {
		if id > w.cursor {
			w.cursor = id
		}
	}
	if headID > w.cursor {
		w.cursor = headID
	}
	w.flush()
	w.logger.Info("authz decision-cache flushed by subject-change poll", "cursor", w.cursor)
}
```

```go
// subject_change_client.go — adapter over InternalIAMServiceClient satisfying
// watcher.Poller. WS-2.3.
package clients

import (
	"context"

	iamv1 "github.com/PRO-Robotech/kacho-proto/gen/go/kacho/cloud/iam/v1"
	"google.golang.org/grpc"
)

type SubjectChangePoller struct {
	client iamv1.InternalIAMServiceClient
}

func NewSubjectChangePoller(cc grpc.ClientConnInterface) *SubjectChangePoller {
	return &SubjectChangePoller{client: iamv1.NewInternalIAMServiceClient(cc)}
}

func (p *SubjectChangePoller) PollSubjectChanges(ctx context.Context, since int64) ([]int64, int64, error) {
	resp, err := p.client.PollSubjectChanges(ctx, &iamv1.PollSubjectChangesRequest{
		SinceId: since, Limit: 1000,
	})
	if err != nil {
		return nil, 0, err
	}
	ids := make([]int64, 0, len(resp.GetChanges()))
	for _, c := range resp.GetChanges() {
		ids = append(ids, c.GetId())
	}
	return ids, resp.GetHeadId(), nil
}
```
