## 3. Root cause analysis

### 3.1 Primary root cause — `_suiteFolderId` is empty

**This is the single root cause for ~95% of VPC and Compute failures.**

The environment file (`environments/local.postman_environment.json`) for both
VPC and Compute contains:

```json
"_suiteFolderId": "",
"_suiteFolderCrossId": ""
```

These two variables are used as `projectId` in nearly every CRUD case's POST
body. When empty:

- The API receives an empty string `projectId`
- kacho-iam authz returns `PermissionDenied` (code 7) because there is no valid
  project scope to check — manifests as HTTP 403
- Or the resource service returns `InvalidArgument` (code 3) because empty id
  fails field validation

**Symptom pattern (from network suite output):**
```
AssertionError: poll status 200
  expected 401 to deeply equal 200
  inside "NET-LIFECYCLE-CONF / poll-op"

AssertionError: status 200
  expected 401 to deeply equal 200
  inside "NET-LIFECYCLE-CONF / lst-excludes"
```

The VPC network suite uses `jwtProjectAdminA1` for CRUD ops. `projectA1Id` IS
seeded in the environment (`prj9j6y00msn65vcfdq3`) but is NOT wired into
`_suiteFolderId`. The authz-deny suite (which uses its own hardcoded project
references) would likely be near-green but was not isolated in the run.

### 3.2 Why `internal-region-zone` is green (77/77)

`internal-region-zone.py` tests `Region.Get`, `Region.List`, `Zone.Get`,
`Zone.List` — all read-only, no `projectId` required. These RPCs are
authenticated but not project-scoped. Hence 100% pass rate.

### 3.3 Why `instance` has 14/50 partial passes

The instance suite has authz-deny-style cases (expecting 403) interspersed
with happy-path cases (expecting 200). The 14 passing assertions are the
ones that expect 401/403 when using `jwtNoBindings` or anonymous tokens —
those fire correctly without a valid projectId. The 36 failures are
happy-path cases that need a real `_suiteFolderId`.

### 3.4 Secondary cause — `internal-cloud` 24/31 fail

`internal-cloud` tests the cloud-level admin scope checks. Some assertions
fail because `existingCloudId` is a legacy pre-KAC-124 value
(`b1g17xaqzkanektj8tz8`) that no longer matches the current IAM model
(Account/Project replaced Organization/Cloud/Folder). This is a fixture
staleness issue independent of `_suiteFolderId`.

### 3.5 `internal-pool` failures (60/69 fail)

AddressPool CRUD requires a seeded pool + project references. Empty
`_suiteFolderId` breaks pool creation requests. Some of the 9 passing
assertions are likely read-only list calls.

## 4. What's needed for parity with IAM

### 4.1 Fixture setup scripts

IAM has `tests/newman/crud-fixture/setup.sh` (creates accounts, projects, users,
mints JWTs) + the full `kacho-workspace/tests/authz-fixtures/setup.sh` (full
authz matrix). VPC and Compute have **neither**.

For VPC/Compute to reach green, the following must be created:

**`kacho-workspace/tests/authz-fixtures/setup-vpc.sh`** (or extend `setup.sh`):

Creates (via REST API against the live stack):
1. A VPC network in project `projectA1Id` → writes `seedNetworkA1Id`
2. A subnet in that network → writes `seedSubnetA1Id`
3. A security group (or uses default-SG from network create)
4. Updates `environments/local.postman_environment.json`:
   - Sets `_suiteFolderId = projectA1Id` (already exists in env but not wired)
   - Sets `_suiteFolderCrossId = projectA2Id` (for cross-project isolation tests)

For Compute:
1. Seeds zone + region catalogue if not already populated (kacho-compute
   InternalZoneService bootstrap)
2. Seeds at minimum one DiskType entry
3. Seeds a base Image for instance creation tests
4. Sets `_suiteFolderId`, `_suiteFolderCrossId`

**The existing `projectA1Id = prj9j6y00msn65vcfdq3`** in both environments is
already seeded by the IAM authz fixture. The simplest fix is a two-line patch
to `environments/local.postman_environment.json`:

```json
"_suiteFolderId": "prj9j6y00msn65vcfdq3",
"_suiteFolderCrossId": "prj1qz01padzd1v9cwry"
```

This is a fast tactical unblock. The full fixture script should be a separate
task.

### 4.2 Compute seed data

Compute suites additionally need:
- At least one `Zone` record (canonical owner is kacho-compute; the `zone.py`
  suite's failures confirm the zones table is empty or not reachable)
- At least one `DiskType` record (24/24 fail on disk-type)
- At least one `Image` record (11/11 fail on image)
- `existingNetworkId`, `existingSubnetId`, `existingSgId` — used by instance
  NIC creation; currently set to placeholder values
  (`enpcomputeseednet001`, `e9bcomputeseedsub001`, `enpcomputeseedsg0001`)
  that don't exist in VPC

### 4.3 Stale fixture values

Both environments still use `existingOrgId`, `existingCloudId`,
`existingProjectId` with pre-KAC-124 resource-manager IDs
(`bpfst7x3jmxhxfa12tgs`, `b1g*`). These should be updated or removed;
`existingProjectId` should point to a live IAM project (`prj*`).

### 4.4 Recommended fix sequence

| Step | Action | Expected impact |
|------|--------|-----------------|
| 1 | Patch `_suiteFolderId`/`_suiteFolderCrossId` in both env files | Unblocks auth → most CRUD cases can fire |
| 2 | Fix `existingProjectId` → live `prj*` value | Unblocks cross-project ref checks |
| 3 | Seed compute catalogue (Zone, DiskType, Image) via k8s job or setup.sh | Unblocks compute read-only + lifecycle suites |
| 4 | Seed VPC seed resources (Network, Subnet, SG) for compute env vars | Unblocks instance NIC creation |
| 5 | Write `setup-vpc.sh` + `setup-compute.sh` analogous to IAM crud-fixture | Long-term: repeatable fixture bootstrap |
| 6 | Fix `internal-cloud` suite expectations for post-KAC-124 Account model | Cleans up `internal-cloud` suite |

## Appendix: VPC internal-region-zone (77/77 GREEN — reference passing suite)

This is the only fully green VPC suite and serves as a reference for what
a passing run looks like. It covers:
- `Region.Get` / `Region.List` (authenticated, no project scope)
- `Zone.Get` / `Zone.List` (authenticated, no project scope)
- NotFound cases with garbage IDs
- Anonymous rejection cases

The fact that it is fully green confirms: the API gateway is reachable, IAM
authn works, kacho-vpc is running, and Region/Zone data is seeded. The failures
elsewhere are fixture-related, not a sign that the services are broken.
