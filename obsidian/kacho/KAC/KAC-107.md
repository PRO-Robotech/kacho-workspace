## Что и зачем

E2-кусок IAM эпика [[KAC-104]]. Заменяет stub `auth_noop` в `kacho-api-gateway` на реальный auth-interceptor, который:

1. Парсит `Authorization: Bearer <jwt>` из gRPC metadata / HTTP header.
2. Валидирует JWT (HMAC-dev режим / Zitadel JWKS production).
3. Через gRPC-direct звонит `kacho-iam.InternalIAMService.LookupSubject` (port 9091, **НЕ через restmux** — loop-prevention запрет #6) для резолва Zitadel `sub` → kacho `usr_…`/`sva_…`.
4. Прокидывает `Principal{Type, ID, DisplayName}` через ctx → x-kacho-principal-* gRPC metadata → backend.
5. Backend через corelib `operations.PrincipalFromContext(ctx)` → `Repo.CreateWithPrincipal(ctx, op, p)` пишет реальный principal в `operations.principal_*` (DoD #5).

**Pragmatic scope cuts (vs full acceptance v1)** — фиксируются как follow-up KAC-тикеты:

- Full Zitadel deploy fix (CrashLoopBackOff на e2c825 rev 8/9) — **follow-up KAC** (chart-config / DB issue). E2 пока работает в `authn.mode = dev` (HMAC-dev токены) + готова прод-ветка кода для Zitadel JWKS когда поправим deploy.
- SA key-flow (`ServiceAccountKey.Create/Get/List/Revoke` + Zitadel management API) — **follow-up KAC**. Core auth flow без SA-keys.
- OIDC `/iam/v1/auth/callback` (cookie session) + UI redirect — **follow-up KAC**. Текущий путь: Bearer-token only.
- `SubjectChangeNotificationService.Watch` listener + NOTIFY publisher — **follow-up KAC**. TTL 30s на subject_cache достаточен.
