## Что и зачем

Закрывает [[KAC-104]] DoD #1 (signup) и фундамент DoD #4 (`Operation.principal` — реальный User/SA вместо stub `system`). До E2 api-gateway пускает любого; `Principal` в `Operation` — stub. После E2:

- **Zitadel** задеплоен в namespace `kacho` (helm chart `zitadel/zitadel`), Postgres `kacho_zitadel`.
- **OIDC-клиент** `kacho-platform` зарегистрирован (PKCE для UI, client-credentials для CI).
- **api-gateway auth-interceptor** (`kacho-corelib/auth/`): валидация JWT (JWKS-cached), вызов `InternalIamService.LookupSubject(externalId) → (subjectType, subjectId, displayName)`, инжект `Principal` в gRPC metadata (`x-kacho-principal-type/id/display-name`).
- **`InternalIamService.LookupSubject`** RPC — internal-only, kacho-iam port 9091; upsert-на-чтении (создаёт User mirror при first-login).
- **`Principal` пробрасывается в `operations.created_by_*`** — backend сервисы (vpc/compute/iam) читают metadata и зовут `corelib/operations.Repo.CreateWithPrincipal`.
- **`auth_noop` удалён** из api-gateway (был E0 stub).
- **SA key-flow** — `ServiceAccount.CreateKey` возвращает private-key one-shot; client-credentials grant в Zitadel за SA.

`Operation.principal` теперь отражает реального user'а ([[KAC-104]] DoD #4 closed). Реактивность permissions (DoD #5) — отдельно в E3 ([[KAC-108]]).

## DoD #1 blocker fix — Zitadel chart deploy (2026-05-17)

**Подход**: гибрид Подходов A + C из brief'а — **two-stage helm install в Makefile / e2c825 README + `wait-for-pg` init-container в `setupJob`/`initJob`** (defense-in-depth).

**Корень бага**: Zitadel chart рендерит `zitadel-init`/`zitadel-setup` как `helm.sh/hook: pre-install,pre-upgrade` Jobs (см. `templates/{init,setup}job.yaml` + defaults в `setupJob.annotations`, hook-weight 1/2). Bitnami `postgresql` sub-chart (`pg-zitadel`) — НЕ hook → создаётся в основной фазе install. Pre-install hook'и выполняются ДО основной фазы → DNS-resolve `kacho-umbrella-pg-zitadel:5432` падает → `BackoffLimitExceeded` → helm upgrade fail.

**Что попробовали**:
- ❌ Чистый Подход A (snять hook через `setupJob.annotations: {}` / `~` в values.yaml) — Helm v3 deep-merge отбрасывает null/empty overlay (helm/helm#9136), chart-default остаётся.
- ❌ Подход A через `--set ...=null` — работает, но прятать критичную семантику в CLI-флагах хуже чем явный two-stage.
- ✅ Подход C (two-stage install) — stage 1 без zitadel (только pg), `kubectl rollout status sts/...-pg-zitadel`, stage 2 с zitadel — pre-install hooks теперь резолвят pg-Service. Реализовано в `Makefile#dev-up` + `clusters/e2c825/README.md`.
- ✅ Bonus: `wait-for-pg` init-container (busybox `nc -z` loop 600s) добавлен в `zitadel.setupJob.initContainers` / `zitadel.initJob.initContainers` — safety-net против pg-Service flap (chart-native поддержка через values).

**Изменённые файлы**:
- `helm/umbrella/values.dev.yaml` — `+47/-1` (zitadel.setupJob/initJob: backoffLimit=5, activeDeadlineSeconds=600, initContainers wait-for-pg)
- `Makefile` — `dev-up` стал two-stage (+15/-1)
- `clusters/e2c825/README.md` — обновлён step 4 (+18/-7)
- `clusters/e2c825/overrides.yaml` — `zitadel.enabled: true`, `pg-zitadel.enabled: true`, обновлён комментарий
