## Что и зачем

Все Kachō-сервисы, производящие Docker-образ, должны собирать multi-arch
(amd64 + arm64) образы и публиковать их в DockerHub. Раньше образы собирались
только локально (`kind load docker-image kacho-*:dev`), не было опубликованных
артефактов для production-деплоя.

Добавлен workflow `docker-build.yml` в 5 репозиториев (kacho-iam, kacho-vpc,
kacho-compute, kacho-api-gateway, kacho-ui) — паттерн адаптирован из
`PRO-Robotech/sgroups` / netguard reference. 3 job'а: amd64 (ubuntu-latest) +
arm64 (self-hosted) + multiarch manifest push. Secrets `DOCKERHUB_USERNAME` /
`DOCKERHUB_TOKEN`. Триггеры — push в `main` / `KAC-*` / тег `v*`.

Polyrepo-специфика: Go-сервисы (iam/vpc/compute/api-gateway) — Dockerfile делает
`COPY kacho-corelib` / `COPY kacho-proto`, поэтому workflow чекаутит siblings в
один каталог (build context = workspace-dir). kacho-ui — self-contained
(context = `.`).

kacho-deploy: umbrella Helm values переключены на DockerHub registry —
`global.imageRegistry` / `global.imageTag` (configurable); per-service `image:`
overrides в `values.yaml` (DockerHub default), `values.dev.yaml` (local `:dev`
образы для kind), `values.prod.yaml` (DockerHub + `${KACHO_IMAGE_TAG}` envsubst).

docker-build.yml дополняет существующие `ci.yaml` (тест/линт), не заменяет.

## Dev-stand re-deploy (kind) — blockers fixed (2026-05-20)

Re-deploy `make dev-up` на kind-кластере `kacho` после M10 Zitadel-removal.
Исходные 4 blocker'а + 6 доп. config/template-дефектов, найденных по ходу:

**Fixed (kacho-deploy, ветка KAC-127):**

- **A — OpenFGA migration deadlock**: `openfga.datastore.migrationType: initContainer`
  (был `job` + `wait-for-migration` init-container → `helm --wait` deadlock).
- **B — kacho-iam wait-for-zitadel**: уже удалён M10-cleanup'ом (verified).
- **C — kacho-ui образ**: `Makefile` target `build-ui` (docker build + kind load),
  вызывается из `dev-up`; `ui` добавлен в `reload-svc`.
- **D — corelib branch mismatch**: worktree `kacho-corelib-KAC-127` удалён,
  `project/kacho-corelib` переключён на ветку `KAC-127` (есть `authz/listobjects.go`).
- **+ third-party image registry**: `global.imageRegistry: ""` на dev — `values.yaml`
  ставил `docker.io/prorobotech`, что ломало Ory/OpenFGA/bitnami образы.
- **+ Ory migration hooks**: `kratos`/`hydra` `automigration.type: initContainer`
  (тот же pre-install-hook deadlock, что у OpenFGA).
- **+ kratosSelfServiceUI**: `enabled: true` — kacho-ui nginx безусловно проксирует
  на этот upstream, при disabled nginx падает.
- **+ OPA image**: `1.0.0-rootless` → `1.0.0-envoy-static` (несуществующий тег;
  нужен opa-envoy для `envoy_ext_authz_grpc` плагина).
- **+ openfga/cli tag**: `0.6.5` → `v0.7.13` (тег без `v`-префикса не существует).
- **+ opaSidecar.iamInternalHost**: `kacho-umbrella-iam-internal` → `kacho-iam-internal`.

**Documented remaining blockers (GitHub-issues, kacho-deploy):**

- **#37** — Hydra config несовместим со схемой `oryd/hydra:v26.2.0` (DPoP/mTLS/
  refresh-rotation keys не в схеме). Phase 2 scope.
- **#38** — `openfga-bootstrap` `dsl-transform` init-container ожидает `/bin/sh`
  в distroless `openfga/cli`. `openfgaBootstrap.enabled: false` (temp). Template-fix.
- **#39** — OPA sidecar не может тянуть bundle (kacho-iam :9091 — gRPC-only, нет
  HTTP bundle-эндпоинта). `opaSidecar.enabled: false` (temp). Phase 3 scope.
