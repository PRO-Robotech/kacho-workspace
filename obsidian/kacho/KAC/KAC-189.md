# KAC-189: kacho-iam — RoleReadAdapter.Get SELECT 10 cols vs scanRole 7 cols

**Status**: done
**Type**: fix
**Repos**: kacho-iam
**PRs**: PRO-Robotech/kacho-iam#44 (merged)
**YT**: https://prorobotech.youtrack.cloud/issue/KAC-189

## Что и зачем

`internal/repo/kacho/pg/role_read_port.go::Get` выполнял:

```sql
SELECT id, cluster_id, organization_id, account_id, project_id,
       name, description, permissions, is_system, created_at
  FROM roles WHERE id = $1
```

(10 cols), но вызывал `scanRole(row)` (`role_repo.go:254`) который сканит в 7 destinations: `id, account_id, name, description, permissions, is_system, created_at`.

pgx возвращал:
> `role read: number of field descriptions must equal number of destinations, got 10 and 7`

→ `MapErr` → `codes.Internal` → HTTP **500**.

## Impact

Все approve-пути минтящие AccessBinding падали 500:
- `JitPendingService.ApproveJITActivation` (зовёт `s.roles.Get` для FGA-tier mapping)
- `Phase7BreakGlassService` (тот же RoleReadPort)
- `Phase7JITService` (auto-grant)
- `Phase7Worker` (expirer rebuild)

В newman: 25 failed assertions в `iam-jit-pending` suite (run 26361394730).

## Fix

Привести SELECT в `role_read_port.go::Get` к 7-col projection scanRole (drop `cluster_id, organization_id, project_id` — их нет в `domain.Role`). Канонический scanRole — `role_repo.go`.
