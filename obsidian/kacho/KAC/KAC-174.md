# KAC-174: kacho-nlb config DSN expand `$(VAR)` placeholder

**Status**: in-progress (PR #17 open, CI running)
**Type**: fix (regression from KAC-141 NLB epic)
**Branch**: `KAC-172-nlb-dsn-env-expand` (имя ветки осталось от первоначального placeholder-номера — YouTrack auto-incremented to 174; PR description / commits / vault — всё за KAC-174)

## Bug

После merge'а KAC-141 (NLB epic) — kacho-nlb pod **stuck in `Init:CrashLoopBackOff(7)`** на CI.

Корневая причина: Helm рендерит `repository.postgres.url` с **shell-style** placeholder `$(KACHO_NLB_DB_PASSWORD)` (password — k8s Secret, не в ConfigMap). Viper **не expand'ит** `$(VAR)` синтаксис — migrator передаёт literal placeholder в pgx → connection fail → init-container CrashLoopBackOff.

Эффект ⇒ **оба** newman-e2e workflow'а (kacho-deploy + kacho-nlb) падают на `bootstrap fixtures + run kacho-iam authz newman suites` step — kacho-nlb stack никогда не поднимается, kacho-iam Operation.Create начинает hang (race condition в setup.sh? — нужно проверить после fix'а deployment'а).

## Fix

1. `internal/apps/kacho/config/config.go`: добавить `PasswordFromEnv string` в `PostgresConfig` (mapstructure `password-from-env`).
2. `internal/apps/kacho/config/load.go`: `expandPasswordFromEnv(&cfg)` после `viper.Unmarshal` — заменяет `$(<name>)` → `os.Getenv(<name>)` в `URL` и `SlaveURL`.
3. `deploy/templates/configmap.yaml`: emits `password-from-env: KACHO_NLB_DB_PASSWORD` рядом с URL.
4. `deploy/templates/deployment.yaml` — без изменений (env-var `KACHO_NLB_DB_PASSWORD` уже прокинут в init+main контейнеры).
