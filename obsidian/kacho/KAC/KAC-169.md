# KAC-169: opsproxy principal metadata propagation

**Status**: test  →  done after merge
**Type**: fix (production gap)
**YT**: https://prorobotech.youtrack.cloud/issue/KAC-169
**PR**: [kacho-api-gateway#31](https://github.com/PRO-Robotech/kacho-api-gateway/pull/31)

## Что и зачем

**Найдено**: live newman 308 RED после W1.5/W1.6. Любой user, поллящий свою же
`Operation.Get(id)` через REST gateway, получает `NotFound` — backend (IAM/VPC/Compute/NLB)
не видит owner-principal.

**Root cause**: `internal/opsproxy/proxy.go::Get` / `Cancel` передавали `ctx` напрямую
в `client.Get(ctx, req)`. `restmux/mux.go::WithMetadata(principalMetadata)` ставит
`x-kacho-principal-*` в **incoming** gRPC metadata. gRPC outgoing нужно собрать
руками — это уже делается в `internal/proxy/director.go:55-56` и
`internal/proxy/shimproxy.go:44-45`. `opsproxy` был единственный hop, где этого
не было.

**Fix**: `propagateMetadata(ctx)` helper — same pattern что в director/shimproxy.
TDD: 3 RED-теста c `principalCapturingServer` mock, читающим incoming
metadata из своей ctx, → 3 GREEN после 2-line fix в Get/Cancel.
