# KAC-128: AccessBinding.Create idempotency — metadata returns existing id on 5-tuple conflict

**Status**: done
**Type**: fix
**Repos**: kacho-iam
**PRs**: (not pushed — local branch KAC-128, commit e0bdeb9)
**YT**: https://prorobotech.youtrack.cloud/issue/KAC-128

## Что и зачем

BUG-7 (acceptance §13.4): при повторном Create с тем же 5-tuple
`(subject_type, subject_id, role_id, resource_type, resource_id)` DB-уровень
корректно возвращал существующий row (ON CONFLICT … DO UPDATE SET id = access_bindings.id),
но Operation-metadata `CreateAccessBindingMetadata.access_binding_id` содержала
КАНДИДАТ-id, сгенерированный до вызова worker'а — не существующий id.

Фикс: добавлен `resolveOrGenerateID(ctx, b)` — вызывает `FindExisting` через Reader
ДО создания Operation. Если 5-tuple уже существует — metadata получает существующий id.
DB-уровень ON CONFLICT остаётся safety-net для TOCTOU-гонок.
