## Что и зачем

**KAC-229**: UI листает листенеры на project-level (`/projects/:id/nlb/listeners` → шлёт `project_id`), но `ListenerService.List` требовал `load_balancer_id` (per-LB), authz scope `lb_network_load_balancer:{load_balancer_id}` → без LB-id → `lb_*:*` → no-path → 403. NLB/TG List уже project-scoped; выровнял Listener List:
- proto: `ListListenersRequest` + `project_id` (required), `load_balancer_id` → optional filter; authz scope object=`project`, from `project_id`.
- nlb use-case: require `project_id` (repo уже умеет ProjectID-фильтр + индекс `(project_id,created_at,id)`).
- gateway catalog: listeners.list → project.

**KAC-227 (nlb-слой, пропущенный в proto/gateway-PR)**: per-RPC authz-interceptor берёт object_type из **хардкод-карты** `internal/check/permission_map.go`, а НЕ из proto. Там было `nlb_load_balancer`/`nlb_listener`/`nlb_target_group`, тогда как FGA-модель + tuple-writes nlb = `lb_*`. Gateway-edge Check пропускал (catalog=lb_*), а backend-interceptor денил по `nlb_load_balancer:<id>` (нет tuple) → per-RPC 403 на NLB/TG/Listener Get. Выровнял константы → `lb_*`.

## Урок (важно)

`unauth→401` проверяет только **gateway-слой**. Backend per-service interceptor (`kacho-corelib/authz` + хардкод `permission_map.go`) срабатывает только для **аутентифицированных** запросов и имеет **свой** object_type (НЕ из proto). При authz-изменениях object_type правится в ТРЁХ местах: proto scope_extractor, gateway embed catalog, **и `permission_map.go` каждого сервиса**. Пересборка сервиса от нового proto НЕ меняет хардкод-карту.
