## Что и зачем

После 34 incremental миграций (включая откатные пары `0011↔0012`, `0016↔0017`,
backfill-шаги `0020`, retrospective ALTER CHECK `0025..0033`, schema rename
`0034`) `internal/migrations/` стал плохо читаемым: чтобы понять текущую
схему — нужно проиграть в голове 34 ALTER'а. Стенд пересобирается с нуля,
production-релизов VPC не было — это идеальный момент свернуть всё в один
чистый baseline.

**Цель**: единый `0001_initial.sql`, который даёт **точно тот же** конечный
state БД, но:
- Таблицы создаются сразу в схеме `kacho_vpc` (skill `evgeniy` §5 E.4) —
  без промежуточного rename из `public`.
- Все CHECK / FK / UNIQUE / EXCLUDE / generated columns / триггеры inline
  с `CREATE TABLE` (skill `evgeniy` §5 E.5).
- Нет backfill-шагов, нет откатных миграций.
- Нет legacy полей kube-ovn / `vpn_id` / NIC data-plane (drop KAC-79/KAC-36).
