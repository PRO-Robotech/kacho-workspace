## Что и зачем

В серии тестов KAC-130 были найдены баги в authz-слое kacho-iam. KAC-131 их фиксит и перемеривает все 12 newman-сьютов на живом kind-стенде.

**BUG-6**: `AccessBinding.Get` возвращал 403 после Create для account-scoped AB.  
Причина: `tuples.go` писал FGA-иерархический тупл `project:<id> → iam_access_binding:<id>` ТОЛЬКО для `ResourceType == "project"`. AB с `ResourceType == "account"` не получал тупл → FGA cascade не разрешался → 403.  
Фикс: `AccessBindingService/{Get,Delete}` → `<exempt>` в permission_catalog.json; handler `get.go` реализует scope-filter сам.

**BUG-8**: `ListByResource`/`ListBySubject` имели неверный `scope_extractor` — `object_type: project`, `from_request_field: resource_id`/`subject_id`. FGA проверялся против случайного поля как project-ID → всегда `no path` → 403.  
Фикс: оба RPC → `<exempt>`; handler-level `RequireAuthenticated` guard добавлен.

**BUG-3 partial**: `Account.Create` с чужим `ownerUserId` возвращал code 7 (PERMISSION_DENIED / 403). По семантике — это InvalidArgument (caller подал невалидный запрос).  
Фикс: `authzguard.RequireOwnerMatchesPrincipal` → `codes.InvalidArgument` (3). Для Project/Group/SA — архитектурная проблема (FGA говорит "no path" для несуществующего accountId → 403 вместо 400). NEEDS-DECISION, не в scope KAC-131.

**BUG-1 (тест-баг)**: `poll_operation_until_done` / `assert_op_error` / `assert_op_success` отправляли запросы без авторизации → 401. Фикс: добавлен параметр `auth="jwtAccountAdminA"`.

**Security fix (get.go/delete.go)**: несуществующий AB → `PermissionDenied` (не `NotFound`) — предотвращает ID enumeration через side-channel.
