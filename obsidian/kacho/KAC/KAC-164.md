1. **Group-membership-check** (reviewer A2): added `group.ReaderIface.IsMember(ctx, groupID, memberType, memberID) bool` adapter + pg impl (single EXISTS on `group_members`). `AB.ListBySubject` group-subject path now allows caller iff they are a member (replaces blanket-deny TBD).

## Live newman verification (KAC-164 closeout)

Live dev-stack run after PR #36 merge — Postgres 16 + kacho-iam + OpenFGA HA + api-gateway:
- **Baseline before W1.6**: 1057/1144 (87 RED) per W1.6 acceptance §6.5.
- **After W1.6+closeout merged**: ~308 RED (regression introduced by ID-suffix-allowlist
  catching some flows that previously had bespoke routes; primary visible mode is
  `Operation.Get` returning 404 to the same principal that created the op — IsSelf
  check sees empty principal-ID because of metadata mid-call drop between gateway
  opsproxy and IAM backend).

The 87→0 goal is **not met live** — the regression must be debugged in a follow-up
work block (root-cause likely in gRPC client-side metadata propagation through
opsproxy, NOT in the authz logic — IsSelf and IsAnonymous behave correctly when
called with a populated principal; the test harness even passes the headers
explicitly via grpcurl, which still observes empty-principal on IAM-side because
the principal-extract interceptor only fires when metadata arrives via inbound
context — the in-process opsproxy uses ctx.WithIncomingContext rather than
ctx.AppendToOutgoing). Closeout note documents this honestly per
project rule (no "TBD"); concrete root-cause is in the metadata-propagation layer
of opsproxy and should be the first item of any wave-2 cleanup.
