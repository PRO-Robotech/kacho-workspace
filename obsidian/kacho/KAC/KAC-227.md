# KAC-227: NLB authz object_type mismatch nlb_* vs FGA lb_* → Get/sub-list 403

**Status**: done
**Type**: fix
**Repos**: kacho-proto, kacho-api-gateway
**PRs**: PRO-Robotech/kacho-proto#43 (merged), PRO-Robotech/kacho-api-gateway#60 (merged)
**YT**: https://prorobotech.youtrack.cloud/issue/KAC-227
**Follow-up**: KAC-178 (NLB)

## Что и зачем

Системный (латентный) баг authz-проводки loadbalancer: `scope_extractor.object_type` в proto + gateway `permission_catalog.json` использовал `nlb_load_balancer` / `nlb_listener` / `nlb_target_group`, тогда как FGA-модель (`iam/v1/fga_model.fga`) и tuple-writes `kacho-nlb` — `lb_network_load_balancer` / `lb_listener` / `lb_target_group`. Per-RPC authz Check спрашивал тип без tuple'ов → **Get NLB / TargetGroup Get / Listener List = 403** (project-scoped List работал, т.к. object_type=`project`). Найдено при ручном тестировании стенда.

## Затронутые сущности vault

- [[../resources/nlb-network-load-balancer]] — FGA object_type = `lb_network_load_balancer`
- [[../rpc/nlb-network-load-balancer-service]] — scope_extractor object_type
- [[../edges/nlb-to-iam-authz]] — authz Check на lb_* типах

## Фикс

object_type выровнен по FGA-модели в 3 loadbalancer service-proto; `buf generate` обновил gen/ (descriptors + `gen/permission_catalog.json`); gateway синкнул embed-catalog. Проверено: FGA Check `viewer` на `lb_network_load_balancer:<id>` / `lb_target_group:<id>` = allowed (creator-admin каскад); unauth REST → 401 (было 403 AUTHZ_DENIED).

## Известный связанный UI-вопрос (отдельно)

Listener List scoped по `load_balancer_id` (per-LB), а UI имеет project-level nav «Listeners» без LB-id → `lb_*:*` no-path. Листенеры должны жить на detail-странице LB. Не входит в этот фикс.
