# KAC-272: AddressPool DB-level CIDR overlap prevention (EXCLUDE gist)

**Status**: test
**Type**: feature
**Repos**: kacho-vpc

**YT**: https://prorobotech.youtrack.cloud/issue/KAC-272

## Что и зачем

`AddressPool.v4_cidr_blocks` / `v6_cidr_blocks` (text[]) могли пересекаться — внутри
пула И между пулами — без валидации. IPAM аллоцирует external-IP из CIDR пула; UNIQUE
`addresses_external_pool_ip_uniq` — per-pool `(address_pool_id, address)`, поэтому два
пула с пересекающимися CIDR аллоцировали ОДИН external-IP дважды (разные pool_id → UNIQUE
проходит). Реальный IPAM-баг.

Решение (запрет #10 — within-service инвариант на DB-уровне): нормализованная child-таблица
`address_pool_cidrs(pool_id FK ON DELETE CASCADE, kind smallint, block cidr, PK(pool_id,block))`
+ `EXCLUDE USING gist (kind WITH =, block inet_ops WITH &&)` (миграция `0004`; declarative,
race-free by construction — зеркалит `subnets_no_overlap_v4`). Scope инварианта = per `kind`
(cross-zone public CIDR глобально непересекающиеся → zone в exclusion-key НЕ входит).

Пересечение на `Create` / `:addCidrBlocks` → SQLSTATE `23P01` → `FailedPrecondition`
«address pool CIDRs can not overlap» (grpc-gateway 400). Sync within-request precheck
(`checkPoolCIDRsDisjoint` через `net/netip`) → `InvalidArgument` тем же текстом (fast-fail);
DB EXCLUDE — backstop для cross-pool/concurrent. `:removeCidrBlocks` удаляет block из
`address_pool_cidrs` → освобождает диапазон.
