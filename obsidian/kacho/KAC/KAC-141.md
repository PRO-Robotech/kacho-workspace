## Что и зачем

Новый сервис `kacho-nlb` — production-ready L4 Network Load Balancer (TCP/UDP) control-plane. Полное покрытие newman тестами, без тех. долга. Единственный TODO — `GlobalLoadBalancer` (cross-region anycast/DNS, reserved slot only, AWS-style composition layer).

## Scope

**In scope**:
- 3 first-class resources: `LoadBalancer`, `Listener`, `TargetGroup` (+ embedded `Target` 4-way oneof + `HealthCheck`)
- 4 public services + 1 internal: NLB / Listener / TargetGroup / Operation / InternalResourceLifecycle
- 30 permission catalog (`loadbalancer.*` namespace) + custom roles flow
- FGA Check + D-11 sync creator-tuple write + D-13 lifecycle subscribe
- VIP allocation: auto via `vpc.InternalAddressService` + opt-in BYO `vpc.Address`
- 4-way Target identity: `instance_id` (compute) / `nic_id` (vpc) / `ip_ref` in-cloud (vpc subnet validate) / `external_ip` out-of-cloud (bogon-check sync)
- 2-phase RemoveTargets drain (deregistration_delay)
- Same-region constraint LB↔TG
- FK RESTRICT cascade — delete bottom-up
- evgeniy §1-§13 compliant: per-resource UseCases, CQRS Repository, DTO table-driven, self-validating domain, viper/YAML config, separate cmd/migrator, ExecAbstract, corlib client-builder

**Out-of-scope (TODO)**:
- `GlobalLoadBalancer` cross-region — proto reserved field-numbers 30-39/10-19 + id prefix `glb` + architecture plan in `docs/architecture/12-future-cross-region.md`. No code in MVP.
