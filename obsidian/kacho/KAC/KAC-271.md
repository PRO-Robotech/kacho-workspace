# KAC-271: per-resource VPC id-prefixes (net/sub/adr/rtb/sgr/gtw/nic/apl)

**Status**: done
**Type**: feature
**Repos**: kacho-corelib, kacho-vpc
**PRs**: PRO-Robotech/kacho-corelib#20, PRO-Robotech/kacho-vpc#139
**YT**: https://prorobotech.youtrack.cloud/issue/KAC-271

## Что и зачем

Раньше VPC-ресурсы делили префиксы: Network/RouteTable/SecurityGroup/Gateway → `enp`,
Subnet/Address/NetworkInterface → `e9b`, AddressPool → литерал `apl`. Тип ресурса не
читался по id. KAC-271 даёт каждому свой 3-char prefix: Network `net`, Subnet `sub`,
Address `adr`, RouteTable `rtb`, SecurityGroup `sgr`, Gateway `gtw`, NetworkInterface
`nic`, AddressPool `apl`.

Routing не ломается: resource-RPC маршрутизируются по REST-path, а не по id-prefix.
`PrefixOperationVPC` **декаплен** от `PrefixNetwork` и зафиксирован как `enp` —
gateway opsproxy маршрутизирует `OperationService.Get` по первым 3 символам op-id,
поэтому op-prefix должен быть стабильным per-домен (а не per-ресурс). Существующие
`enp…`-операции в БД продолжают роутиться.

`corelib/validate.resourceIDPrefixes` расширен новыми префиксами; старые `enp`/`e9b`
(+ legacy `b1g`/`bpf`, compute `epd`/`fd8`) остаются known — проверка family-agnostic,
не должна давать InvalidArgument на корректные id переходного периода.
