# KAC-269: AddressPool CIDR-управление как у Subnet

**Status**: in-progress
**Type**: feature
**Repos**: kacho-proto, kacho-vpc
**YT**: https://prorobotech.youtrack.cloud/issue/KAC-269

## Что и зачем

Раньше CIDR-состав AddressPool менялся через `Update` с replace-флагами
(`replace_v4/v6_cidr_blocks`) — implicit full-replace, без use-guard'а, с
freelist-rebuild на каждый Update. KAC-269 приводит AddressPool к той же модели,
что и Subnet: отдельные RPC `:addCidrBlocks` / `:removeCidrBlocks`, а `Update`
перестаёт трогать CIDR вообще.

- **proto** (ветка KAC-269): + `InternalAddressPoolService.AddCidrBlocks` /
  `RemoveCidrBlocks` (`Add/RemoveAddressPoolCidrBlocksRequest`, REST
  `:addCidrBlocks`/`:removeCidrBlocks`); из `UpdateAddressPoolRequest` удалены
  `v4/v6_cidr_blocks` + `replace_v4/v6_cidr_blocks`.
- **kacho-vpc**: Update без CIDR; sync use-case'ы add/remove + handler; repo-методы
  `AddCidrToFreelist` / `DeleteFreelistForCidrs` / `CountAllocatedInCidrs`.

## Реализация

- **AddCidrBlocks** (`add_cidr_blocks.go`): family-strict + host-bits валидация →
  Writer-TX: Get → append+dedup → Update → `AddCidrToFreelist(newV4)` (только
  дельта) → `InitIPv6PoolCursor` если v6 впервые → outbox UPDATED.
- **RemoveCidrBlocks** (`remove_cidr_blocks.go`): Get → subtract (not-found →
  FailedPrecondition) → empty-guard (InvalidArgument) → `DeleteFreelistForCidrs`
  (row-lock) → `CountAllocatedInCidrs` use-guard (allocated → FailedPrecondition
  "address pool CIDR <cidr> has allocated addresses") → Update → outbox.
- **Атомарность remove vs alloc**: DELETE free_ips первым берёт row-lock —
  конкурентный `AllocateIPFromFreelist` (`FOR UPDATE SKIP LOCKED`) либо успел
  (его IP в addresses → use-check >0 → abort), либо ждёт (после remove free_ip
  нет → не возьмёт). Всё в одной writer-TX.
