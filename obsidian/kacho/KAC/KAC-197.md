## Что и зачем

Завершение Federation-трилогии. После 3a (private_key_jwt SA) и 3b (Federation IN: external IdP → kacho), нужен симметричный путь **OUT**: токены, выпущенные нашей Hydra, должны быть пригодны для аутентификации в **внешних** IAM (AWS STS AssumeRoleWithWebIdentity, GCP Workload Identity Federation, Azure AD federated credentials).

Технически — публичный OIDC discovery (`/.well-known/openid-configuration` + JWKS), стабильный `iss`, корректный `aud`/`sub`, доступный через public TLS endpoint.

## Что делаем

1. **Hydra public ingress + TLS** — `helm/umbrella/values.{dev,prod}.yaml`:
   - Hydra public listener exposed как `auth.kacho.local` (dev) / реальный домен (prod)
   - Cert-manager `Issuer` для TLS (dev: self-signed; prod: Let's Encrypt placeholder)
   - `hydra.config.urls.self.issuer` = публичный URL (стабильный `iss` claim)
2. **Audience в IssueSAKey** — proto+API:
   - `IssueSAKeyRequest.audience repeated string` (опционально)
   - Mapping в Hydra-client `audience` field → токены получают `aud` claim
3. **Docs**: `kacho-iam/docs/components/22-federation-out.md`
   - Sequence diagram: kacho-SA → kacho-Hydra-token → AWS STS
   - Tutorial: AWS (audience = `sts.amazonaws.com`)
   - Tutorial: GCP Workload Identity Pool (audience = `//iam.googleapis.com/projects/...`)
   - Security: stable `iss`, JWKS rotation, audience pinning
4. **Tests**: Go-integration — IssueSAKey с audience → check Hydra-client запоминает audience.
