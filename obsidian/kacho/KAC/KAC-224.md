## Что и зачем

Аудит адаптации kacho-ui под IAM-изменения: UI догнал items 1-5 / KAC-196, но не RBAC v2 (KAC-214..218). Закрыты два реальных, верифицируемых UI-пробела:

1. **scope read-only колонка.** RBAC v2 добавил `AccessBinding.scope` (CLUSTER/ACCOUNT/PROJECT, proto field 15, **output-only** — в `CreateAccessBindingRequest` поля scope нет, backend derive из `resource_type` через `access_bindings_scope_default_trg`; `dto/toproto/access_binding.go` эмитит на каждом ответе). UI-тип `AccessBinding` его не имел. Добавлены `Scope` type + `scope?` поле + колонка.
2. **Удаление мёртвых resource_type.** `RESOURCE_TYPES` + `api/iam.ts ResourceType` предлагали `folder`/`organization`/`cloud` (resource-manager era; удалены KAC-124, дропнуты [[KAC-223]] mig0008). Backend `validResourceTypes` их не содержит → `INVALID_ARGUMENT "Illegal argument resource_type"`. Оставлено `account`/`project`/`cluster`.

## Out of scope — backend finding

Permissions-грамматика **3-seg** (domain Go `permissionElementRe`, `types.go:106`) vs **4-seg** (DB CHECK `iam_permissions_valid` mig0005) — backend-баг kacho-iam: create custom Role сломан на RPC-пути (4-seg рубит domain, 3-seg рубит DB-CHECK). → GitHub Issue **PRO-Robotech/kacho-iam#97** (`bug` + `verified-by:test`). UI `RolesPage` regex намеренно НЕ тронут (бессмыслен без backend-фикса). Blast radius: `authzmap.PermissionsToRelations` принимает 3-seg.
