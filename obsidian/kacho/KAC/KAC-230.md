## Что и зачем

У TargetGroup в UI настраивался `health_check`, но не было управления **targets** — самими бэкенд-эндпоинтами. Backend умел `AddTargets`/`RemoveTargets` (POST `:addTargets`/`:removeTargets` → Operation) + `Target.oneof identity` (`instance_id` | `nic_id` | `ip_ref{subnet_id,address}` | `external_ip{address,zone_id}`) + `weight`; `TargetGroup.targets` (field 9) уже в Get-ответе. Только UI не показывал/не вызывал.

## Реализация (UI, бэкенд готов; все 4 identity)

`TargetGroupDetailPage` — переиспользует `ResourceDetailPage` (общая инфа + health_check) + кастомная секция «Targets» через `secondaryActions`:
- список `targets[]` (рендер по типу identity + weight) + per-row Remove (`:removeTargets`, payload = identity-only, backend матчит по identity);
- «Добавить target» modal: дискриминатор identity (Instance/NIC/in-cloud IP/external IP) + RefSelect (compute-instances / network-interfaces / subnets / zones) + weight → `:addTargets`;
- Operation-polling через `OperationDialog`. Паттерн — `InstanceDetailPage` `:attachDisk`/`:detachDisk`.
- routing: `target-groups` detail → `TargetGroupDetailPage` (остальные NLB-ресурсы — generic).

Экспортируемые чистые хелперы `buildTargetPayload`/`targetIdentity`/`targetIdentityOnly` покрыты unit-тестом (wire-формат oneof — баг-опасное место).
