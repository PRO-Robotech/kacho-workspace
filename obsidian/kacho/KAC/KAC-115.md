# KAC-115: Migrate Zitadel + OpenFGA → Ory stack

**Status**: test (6/6 Playwright pass, ждёт wrap-up commit + KAC-116 follow-up)
**Type**: refactor (IdP/OIDC + AuthZ engine replacement)
**Parent epic**: [[KAC-104]]
**YT**: https://prorobotech.youtrack.cloud/issue/KAC-115

## Что и зачем

Заменили Zitadel (IdP/OIDC) + OpenFGA (REBAC) на Ory-стэк:

- **Kratos** — identities + self-service registration/login/recovery flows.
- **Hydra** — pure OAuth2/OIDC provider (UI делегирует Kratos).
- **Keto** — RBAC/REBAC (Zanzibar-style relation tuples).
- **kratos-selfservice-ui-node** — готовый UI для registration/login/etc.
- **MailHog** — SMTP catch-all для Kratos courier (recovery/verification).

Причина: Zitadel требовал manual SMTP setup для регистрации (блокер «Активация учётной
записи» на 1ое registration), и его console-UI конфликтовал с Kachō-style flow.
Ory-стэк даёт явное разделение AuthN/AuthZ + готовый UI + offline-friendly dev mode.

## Архитектура (после миграции)

```
SPA (5.35.93.58)
  ├─ /login,/registration,/recovery,...   → kratos-selfservice-ui-node:3000
  ├─ /.ory/kratos/public/*                → kratos-public:80 (Kachō nginx proxy_pass)
  ├─ /.ory/hydra/public/*                 → hydra-public:4444
  └─ /iam/v1/*, /vpc/v1/*, ...            → api-gateway:8080
       (cookie ory_kratos_session — для auth)
```

OIDC discovery: `http://5.35.93.58/.ory/hydra/public/.well-known/openid-configuration`
(issuer `http://5.35.93.58/.ory/hydra/public/`).

## Известные tech-долги (→ KAC-116)

1. **kacho-iam AccessBindingService** — пишет в OpenFGA outbox (старый). Переключить
   на Keto: `POST /relation-tuples` + `DELETE /relation-tuples`.
2. **kacho-vpc / kacho-compute Check-interceptor** — зовёт OpenFGA `/check`.
   Переключить на Keto: `POST /relation-tuples/check`.
3. **kacho-api-gateway auth-interceptor** — извлекает Zitadel JWT из `Authorization`.
   Переключить на Kratos session: `GET /.ory/kratos/public/sessions/whoami`,
   взять `identity.id` как `principal_id`.
4. **DoD#4 real principal в Operation** — `principal_id` приходит из шага 3.
   Без него Operation.metadata.principal_id остаётся стабом.
5. **DoD#5 реактивность ≤10s** — после `Keto write` через ≤10s next `Check` видит
   новую relation. У Keto нет outbox-промежуточного слоя; запись напрямую через
   tuple-API → `Check` сразу видит (натурально ≤1s). Проверить тестом.
