# KAC-108: E3 — OpenFGA REBAC + Check-interceptor + реактивность

**Status**: in-progress
**Type**: feature (sub-эпик E3 эпика [[KAC-104]])
**Acceptance**: `docs/specs/sub-phase-2.0-iam-E3-openfga-rebac-acceptance.md` (DRAFT v2, 26 GWT, treated as APPROVED in this session — see §11 RESOLVED Q1/Q2/Q4)
**YT**: https://prorobotech.youtrack.cloud/issue/KAC-108

## Что и зачем

Закрывает [[KAC-104]] DoD #5: «Изменение прав применяется реактивно (≤10s, без перезапуска, без многоминутного TTL)».

Pipeline:

1. **OpenFGA full DSL** (overview §4) поверх stub-модели E0 через `openfga-bootstrap-job` (post-install hook).
2. **AccessBindingService.Upsert/Delete** транзакционно пишет (a) row в `access_bindings`, (b) row в `kacho_iam.outbox` (FGA-write event), (c) row в `kacho_iam.subject_change_outbox` — одна TX.
3. **kacho-iam fga_tuple_writer worker** — drain outbox → openfga.Write / Delete; idempotent (409 = success).
4. **kacho-iam subject_change_notifier worker** — drain subject_change_outbox → `pg_notify('kacho_iam_subjects', subject_id)`.
5. **kacho-iam resource_lifecycle_subscriber worker (D-13)** — subscribe на новый Internal `InternalResourceLifecycleService.Subscribe` (server-stream) каждого peer (vpc/compute/lb); пишет parent-tuples в outbox.
6. **per-service Check-interceptor** в vpc/compute/lb/iam (corelib `authz/`) — на каждом write-RPC: lookup `(object_type, relation, extractor)` в `metadata.PermissionMap` → `InternalIamService.Check`. Cache 5s TTL positive-only + LISTEN-invalidate.
7. **D-11 creator-tuple sync write** — Create handler синхронно пишет own-creator tuple до `tx.Commit()` (≤10ms; устраняет create-then-get race).
8. **D-6 fail-modes** — FGA-unavailable → fail-closed `PermissionDenied`; `KACHO_<SVC>_AUTHZ__BREAKGLASS=true` (dev-only).

Реактивность ≤10s гарантирована: TTL=5s + NOTIFY-invalidate ≤1s + outbox-drain ≤2s + buffer.
