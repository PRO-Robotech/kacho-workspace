# KAC-113: E0 follow-up — sync principal_* в legacy сервисах

**Status**: in-progress

## Что и зачем

E0 PR-1 расширил `kacho-corelib/operations.Operation` (и schema `operations` таблицы) полями
`principal_type / principal_id / principal_display_name` с E0-stub defaults
(`'system'` / `'bootstrap'` / `'kacho-iam-bootstrap'`). В **kacho-iam** новая схема работает
сразу — она greenfield baseline. В **существующих** сервисах (kacho-vpc / kacho-compute /
kacho-resource-manager / kacho-loadbalancer) есть свои `operations`-таблицы в локальных схемах
(`kacho_vpc.operations`, `kacho_compute.operations`, ...) — их надо **синхронизировать**.

Этот тикет — раскатать миграцию `ALTER TABLE operations ADD COLUMN principal_type ... DEFAULT 'system' ...` по 4 сервисам через `make sync-migrations` corelib-flow:

1. **kacho-corelib** — добавить shared миграцию в `kacho-corelib/migrations/common/` (`000X_operations_principal.sql`).
2. Каждый сервисный репо запускает `make sync-migrations` → копирует файл в свой `internal/migrations/`.
3. Каждый сервис обновляет свой `operations.go` repo (если он генерирует INSERT-statement руками) — добавить новые колонки в INSERT (либо опереться на DEFAULT).
4. Worker baggage — `kacho-corelib/operations.Worker` уже умеет писать principal-поля из ctx baggage (PR-1 уже реализовал); сервисам достаточно sync-migration + перезапуск.
