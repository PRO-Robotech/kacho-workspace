## Что и зачем

После E1 (peers переключены на kacho-iam.Project) и E4 (UI больше не использует Organization/Cloud/Folder pages) — `kacho-resource-manager` становится мёртвым кодом: ни один peer его не зовёт, ни одна страница UI его не открывает. E5 финализирует deprecation:

- Helm chart `kacho-resource-manager` **удалён** из `kacho-deploy/charts/` + clusters/e2c825 overrides очищены.
- API-gateway **удаляет registration** для `rm.v1.{Organization,Cloud,Folder}Service.*` (включая internal-mux).
- Postgres database `kacho_rm` дропнут на cluster e2c825 (после dump-а в S3-bucket для disaster-recovery).
- proto-stubs `kacho.cloud.resourcemanager.v1.*` **помечены deprecated** (комментарий `// Deprecated: use kacho.cloud.iam.v1.*`), но **не удалены** из `kacho-proto` (поломает несвязанных подписчиков; clean-removal — workspace-wide refactor под релизную фазу).
- Репозиторий `PRO-Robotech/kacho-resource-manager` помечен **archived** на GitHub (read-only).
- Vault edges `compute-to-rm-folder-check`, `vpc-to-rm-folder-exists`, `apigw-to-rm` — помечены deprecated (статус-frontmatter), описание дополнено «replaced by [[edges/*-to-iam-project-exists]]».
- Workspace docs (`CLAUDE.md`, `docs/specs/01-architecture-and-services.md`) — раздел про RM переписан в формате «historical: до KAC-104, см. iam-эпик».

> [!warning] Этап финализирует ликвидацию: после merge'а PR-chain'а E5 откатиться к RM == восстанавливать БД из dump + revert трёх PR'ов. До тех пор RM остаётся live параллельно с iam — это допускается во время E1..E4.
